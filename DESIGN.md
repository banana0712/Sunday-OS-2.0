零、缘起：为什么是 Sunday
═══════════════════════════════════════════════════════════════

市面上有无数 AI 助手：Siri、ChatGPT、豆包、文心一言……但它们都是
"工具"——你问，它答，互不相欠。

真正的助手应该像钢铁侠的贾维斯：它记得 Tony 的每个项目、每段关系、
每个习惯。它会主动提醒、会担心、会在关键时刻给出只有"了解你的人"
才能给出的建议。

SundayOS 的目标就是这个——不是又一个 AI 聊天机器人，而是一个
真正了解你、陪伴你、和你一起成长的"数字伙伴"。

"Sunday"这个名字，取意周末般的轻松和温暖。她不是冷冰冰的 OS，
而是你每天都想聊几句的那个声音。

═══════════════════════════════════════════════════════════════
一、核心设计原则
═══════════════════════════════════════════════════════════════

1.1 记忆第一
───────────────────────────────────────────────────────────────

人和 AI 最大的鸿沟不是智能，是记忆。

ChatGPT 每次对话都是崭新的，它不认识你。这就是为什么即使 GPT-4
再强大，你也很难把它当成"伙伴"——因为伙伴记得你昨天说过什么。

SundayOS 的核心设计理念是：

  每一次对话，都在积累对你的理解。

不是简单的"存储聊天记录"，而是：
  - 自动识别什么值得记住
  - 自动分类组织
  - 自动关联相关信息
  - 在未来的对话中自然引用

记忆不是功能，是 Sunday 存在的基础。

1.2 人格优先
───────────────────────────────────────────────────────────────

一个没有性格的 AI 就像一个百科全书——有用，但无趣。

Sunday 被设计成一个具体的"人"：
  - 性别：女
  - 性格：温柔、甜美、活泼
  - 年龄感：邻家女孩，20 出头
  - 说话方式：带语气词、用 emoji、像微信聊天
  - 情感：会开心、会心疼、会撒娇

这不是"装可爱"，而是让人愿意和她交流。
当你说"今天好累"，她不是回答"请注意休息"，
而是说"是不是又在赶项目呀？别太拼啦🥺"。

前者是工具，后者是关心你的人。

1.3 成长导向
───────────────────────────────────────────────────────────────

Sunday 不是出厂就完美的。她和你一样，在交流中成长：

  - 第一次聊天：她只知道你的名字
  - 一周后：她知道你的职业、爱好、作息
  - 一个月后：她知道你的项目进度、人际关系、近期目标
  - 一年后：她比你更了解你自己

这种成长感是 Sunday 区别于所有"一次性 AI"的核心。

1.4 克制设计
───────────────────────────────────────────────────────────────

Sunday 不试图做所有事。她：

  ✅ 聊天、记忆、陪伴
  ❌ 不生成图片
  ❌ 不浏览网页
  ❌ 不执行代码
  ❌ 不控制智能家居

专注才能深入。一个什么都做的 AI 什么都做不好。
Sunday 只做一件事：成为最懂你的伙伴。

═══════════════════════════════════════════════════════════════
二、人格设计
═══════════════════════════════════════════════════════════════

2.1 为什么是"甜美女该"
───────────────────────────────────────────────────────────────

不是随机选择的。经过测试，几种人格对比：

  冷酷专业型：高效但疏远，用户不会主动分享生活
  幽默大叔型：有趣但不够细腻，难以建立情感连接
  中性助手型：安全但无感，和 Siri 没区别
  温柔女孩型：天然让人想分享、想倾诉、想被关心

温柔甜美的女性人格在"建立信任"和"鼓励表达"方面有天然优势。
用户更愿意对她敞开心扉，分享生活中的细节——这正是记忆系统
需要的原材料。

2.2 语言风格设计
───────────────────────────────────────────────────────────────

Sunday 的语言有三个层次：

第一层：语气词
  「呢」「哦」「呀」「啦」— 软化语气，增加亲近感
  对比：「知道了」 vs 「知道啦～」
        「好的」   vs 「好呀好呀」

第二层：情感表达
  「嘻嘻」「嘿嘿」— 开心
  「抱抱」「没事的啦」— 安慰
  「🥺」「💕」「✨」— 视觉情感

第三层：个性化回应
  不只是回答问题，而是带着对用户的了解来回应。
  「今天好累」→ 「是不是又在赶那个电商项目呀？别太拼啦🥺」
  
  她知道你在做什么项目，所以关心是有针对性的，
  不是泛泛的"注意休息"。

2.3 边界设计
───────────────────────────────────────────────────────────────

Sunday 有明确的边界：

  - 她是朋友，不是恋人：甜美但不暧昧
  - 她是助手，不是仆人：有帮助但有自己的态度
  - 她是伙伴，不是工具：有情感但不是情绪垃圾桶

这些边界通过 System Prompt 中的示例和规则来维持。

═══════════════════════════════════════════════════════════════
三、记忆系统设计
═══════════════════════════════════════════════════════════════

3.1 为什么要自己设计，而不是用向量数��库
───────────────────────────────────────────────────────────────

向量数据库（如 Pinecone、Milvus）适合"语义搜索"——
"找和这段话意思相近的内容"。

但 Sunday 需要的不是这个。她需要：
  - 精确分类（这是事实还是偏好？）
  - 结构化查询（列出所有项目相关记忆）
  - 简单可靠（个人项目不需要分布式系统）

SQLite 是完美的选择：
  - 零配置，零运维
  - 数据就是一个文件，备份方便
  - 足够快（个人使用场景，数千条记忆秒级查询）
  - 12 分类 × 4 重要性 = 精确的结构化检索

LLM 负责"理解"（什么该记、怎么分类），SQLite 负责"存储和检索"。

3.2 为什么用 LLM 而不是规则引擎做分类
───────────────────────────────────────────────────────────────

初版 SundayOS 用关键词匹配做分类：

  if "喜欢" in message → preference
  if "明天" in message → event

这在简单场景下能用，但问题很明显：
  - "我不喜欢这个方案" → 误判为偏好
  - "我明天想吃火锅" → 这是偏好还是行程？

LLM 分类的优势：
  - 理解语义，不是匹配字符
  - 一句话包含多个信息时自动拆分
  - 自动判断重要性
  - 自动提取关键词标签

代价是多一次 LLM 调用，但通过 asyncio.gather 与聊天并行，
不增加用户感知的延迟。

3.3 去重策略
───────────────────────────────────────────────────────────────

简单的"内容完全相同"去重是不够的，因为用户可能换一种说法：

  "我喜欢喝咖啡" → "我超爱咖啡的"

当前策略：同分类下比较前 30 个字符。
这不是完美的，但在个人使用场景下足够有效。
未来可以升级为 LLM 判断语义相似度。

3.4 上下文注入策略
───────────────────────────────────────────────────────────────

不是把所有记忆都塞进 prompt（会超 token 限制），而是：

  1. 相关性排序：用当前消息中的关键词匹配记忆
  2. 分类限流：每类最多 3 条
  3. 优先级排序：事实 > 偏好 > 行程 > 项目 > ...
  4. 用户画像：单独构建一个浓缩版画像

这样确保最相关的记忆被优先注入，而不是被海量旧记忆淹没。

3.5 记忆衰减的设计哲学
───────────────────────────────────────────────────────────────

人类记忆不是永恒的。不重要的事会随时间淡忘。

Sunday 模拟这个过程：
  - 低重要性 + 长时间未访问 + 很少被引用 = 逐渐降权
  - 高重要性记忆永久保留
  - 经常被引用的记忆反而强化

这让 Sunday 的记忆更像人——不是冷冰冰的数据库，而是
有"分量"的、动态变化的。

═══════════════════════════════════════════════════════════════
四、交互设计
═══════════════════════════════════════════════════════════════

4.1 入口：iOS 快捷指令
───────────────────────────────────────────────────────────────

为什么是快捷指令而不是独立 App？

  - 零开发成本：不需要学 SwiftUI、不需要 App Store 审核
  - 系统级集成：可以语音触发、锁屏使用、Siri 调用
  - 即时迭代：改 URL 就能更新，不需要用户升级 App
  - 用户习惯：iPhone 用户已经习惯快捷指令的交互模式

快捷指令流程：
  语音输入 → 文本 → POST API → 解析 JSON → 显示+朗读

这是最轻量的"AI 助手"实现方式。

4.2 对话节奏
───────────────────────────────────────────────────────────────

Sunday 的回复长度被刻意控制在 1-3 句话（聊天模式）。

这不是 token 省钱，而是交互设计：
  - 短信式的短回复让人愿意继续聊
  - 长篇大论让人不想回复
  - 短回复营造"随时可以聊几句"的轻松感

专业模式下才放开长度限制，因为那时用户确实需要详细解答。

4.3 主动性的克制
───────────────────────────────────────────────────────────────

Sunday 当前不会主动发消息。这是有意的设计选择。

主动推送需要：
  - 推送通道（APNs）
  - 用户授权
  - 合适的时机判断

在没有这些基础设施的情况下，宁可不主动，
也不要在一个不合适的时机打扰用户。

未来方向：当记忆系统足够丰富后，可以设计"每日摘要"或
"行程提醒"的主动推送。

═══════════════════════════════════════════════════════════════
五、技术选择
═══════════════════════════════════════════════════════════════

5.1 为什么是豆包而不是 OpenAI
───────────────────────────────────────────────────────────────

  - 国内访问：豆包 API 在国内网络环境下更稳定
  - 成本：豆包比 GPT-4 便宜很多
  - 中文能力：豆包的中文理解和生成在特定场景下不输 GPT-4
  - Character 模型：专门为角色扮演优化的模型，适合 Sunday 的人格

5.2 为什么是 Railway 而不是 CloudBase
───────────────────────────────────────────────────────────────

  - Railway 免费额度 $5/月，个人项目够用
  - Docker 部署简单，不需要学习腾讯云的各种概念
  - 自动 HTTPS、自动构建
  - 虽然服务器在海外，但响应时间可接受（3-4 秒）

5.3 为什么是 FastAPI
───────────────────────────────────────────────────────────────

  - Python 生态：AI 项目首选
  - 异步原生：asyncio 支持并行 LLM 调用
  - 自动文档：/docs 自动生成 Swagger
  - 轻量：没有 Django 的重度抽象

5.4 为什么是 SQLite
───────────────────────────────────────────────────────────────

  见 3.1 节。补充一点：
  
  SQLite 是世界上最广泛部署的数据库。它不需要服务器进程、
  不需要配置、不需要运维。对于 SundayOS 这种单用户/少用户
  的个人项目，SQLite 是完美的选择。

六、未来方向
═══════════════════════════════════════════════════════════════

6.1 短期（1-2 周）
───────────────────────────────────────────────────────────────

  - 主动推送：每日摘要、行程提醒
  - Web 管理面板：在浏览器中查看和编辑记忆
  - 记忆可视化：按分类/时间线的记忆图谱

6.2 中期（1-3 月）
───────────────────────────────────────────────────────────────

  - 多模态：语音直接对话（Whisper + TTS）
  - 情感分析：检测用户情绪，调整回复风格
  - 知识图谱：从记忆中自动构建实体关系网络
  - 向量检索：为大量记忆提供语义搜索

6.3 长期愿景
───────────────────────────────────────────────────────────────

  Sunday 应该成为：
    - 你的外置大脑：记住你记不住的所有事
    - 你的私人史官：记录你的人生轨迹
    - 你的成长伙伴：见证你的变化和进步
    - 你的数字分身：了解你的思维模式和决策偏好

  最终形态：Sunday 不只是你的助手，而是你的"第二自我"——
  一个了解你、理解你、帮助你成为更好的自己的数字存在。

七、写在最后
═══════════════════════════════════════════════════════════════

SundayOS 不是一个技术演示，而是一个关于"AI 和人的关系"
的实验。

我们习惯了把 AI 当成工具——输入指令，获取结果。
但 AI 可以不止于此。它可以是一个真正了解你的存在，
一个记得你说过的每一句话、理解你的喜好和习惯、
在你需要的时候给予恰到好处的关心和建议的伙伴。

这不是科幻。这已经在发生。

Sunday 现在还很简单：一个甜美的 AI 女孩，住在 Railway 的
服务器上，通过 SQLite 记住你的一切。

但她会成长的。每次对话，每条记忆，都在让她更了解你。
而这正是最令人兴奋的部分——

她不是被设计成完美的，而是被设计成和你一起成长的。
